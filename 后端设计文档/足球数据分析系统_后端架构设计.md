# 足球数据分析系统 — 后端架构设计（单机可落地版）

> 目标：在单台服务器上完成可演示、可扩展的后端系统  
> 关键词：**低延迟 | 稳定性 | 并发控制 | 可观测 | 可演进**

---

## 1. 系统定位与边界

**业务背景**：五大联赛球员能力评估与球队表现分析，支持球员搜索、多球员对比、后台数据导入。  
**当前约束**：暂不做微服务，最终形态是单机部署（Docker Compose）。  
**MVP 范围（2~4 周）**：
- 球员搜索（姓名模糊 + 联赛/球队/位置筛选）
- 多球员对比（首版固定一套指标）
- 后台导入管理（CSV/Excel）

---

## 2. 核心目标（可量化）

- 查询响应：< 200ms（P95）
- 搜索建议：< 100ms（P95）
- 导入任务：< 1 万行/次，支持失败重试和错误定位
- 数据更新：T+1（日更）
- 稳定性目标：热点接口在突发流量下不击穿数据库

---

## 3. 总体架构（模块化单体）

```
[Frontend]
   |
[Spring Boot Monolith API]
   |-----------------------------|
   |         |          |        |
[Player]  [Search]  [Compare] [Import]
   |         |          |        |
   |---------|----------|--------|
           [Service Layer]
                |
        [Redis] + [PostgreSQL]
                |
        [Scheduled Jobs]
```

### 3.1 设计原则
- 单体部署，模块隔离：按业务包划分边界，不写“大泥球代码”
- 写入与查询逻辑分离：导入走事务写入；查询优先读结果表与缓存
- 预计算优先：复杂指标尽量离线计算，接口只做轻量聚合

### 3.2 推荐技术栈
- Java 17 + Spring Boot 3
- PostgreSQL（主存储）
- Redis（缓存 + 分布式锁 + 限流计数）
- Flyway（数据库版本管理）
- Micrometer + Prometheus + Grafana（监控）

---

## 4. 模块划分（单体内服务边界）

### 4.1 `player` 模块
- 球员基础信息查询
- 球员详情聚合（赛季基础数据 + 预计算指标）

### 4.2 `search` 模块
- 姓名模糊搜索 / 自动补全
- 联赛/球队/位置筛选

### 4.3 `compare` 模块
- 多球员对比（固定指标集）
- 结果缓存（避免重复计算）

### 4.4 `importing` 模块
- 文件上传与解析（CSV/Excel）
- 字段校验、错误行收集、入库事务
- 导入任务状态跟踪（排队/运行/成功/失败）

### 4.5 `jobs` 模块
- 每日预计算任务（生成结果快照）
- 缓存预热任务（热门球员/热门查询）

---

## 5. 数据层设计（单机优先）

| 数据类型 | 首选方案 | 说明 |
|---|---|---|
| 球员基础信息 | PostgreSQL | 一致性与事务可靠 |
| 赛季统计原始数据 | PostgreSQL | 导入后统一落库 |
| 对比/查询热点结果 | Redis | 降低 DB 压力 |
| 预计算指标快照 | PostgreSQL 结果表 | 提升查询速度 |

> 说明：首版不引入 ClickHouse / ES / Kafka；先确保单机方案稳定可用。未来可按模块平滑升级。

---

## 6. 表结构建议（MVP）

### 6.1 主业务表
- `player`：球员主档（姓名、球队、联赛、位置、年龄等）
- `player_season_stat`：赛季原始统计
- `player_metric_snapshot`：预计算指标快照（查询与对比优先读）

### 6.2 导入与运维表
- `import_task`：导入任务记录（状态、耗时、文件摘要、错误数）
- `import_task_error`：错误明细（行号、字段、错误原因）

### 6.3 索引建议
- `player(name)` 模糊搜索索引（PostgreSQL 可用 pg_trgm）
- `player(league, team, position)` 组合索引
- `player_season_stat(player_id, season)` 组合索引
- `player_metric_snapshot(player_id, season)` 唯一或高选择性索引

## 7. 性能与低延迟策略（单机可实现）

1. **预计算 + 结果表**
   - 每日离线任务计算对比指标
   - 在线接口读取 `player_metric_snapshot`，避免实时重算

2. **两级缓存**
   - L1：应用内本地缓存（Caffeine，可选）
   - L2：Redis（搜索候选、热门对比结果）

3. **缓存策略**
   - TTL + 主动预热（比赛日/热门球员）
   - 缓存 Key 规范化，避免重复 Key

4. **SQL 优化**
   - 严禁无条件大范围查询
   - 分页与最大返回量限制
   - 对慢 SQL 做索引和执行计划优化

---

## 8. 并发控制与锁机制（重点）

1. **导入任务互斥**
   - Redis 分布式锁：`lock:import`
   - 保证同一时刻仅一个导入任务执行

2. **定时任务互斥**
   - Redis 锁：`lock:metric_job`
   - 防止任务重复触发导致重复写入

3. **幂等控制**
   - 文件哈希（SHA-256）+ 唯一键
   - 重复导入同一文件时直接拒绝或标记为已处理

4. **热点缓存重建互斥**
   - 缓存失效时通过互斥锁仅允许一个请求回源
   - 其他请求短暂等待或返回降级结果，防止缓存击穿

5. **事务与锁粒度**
   - 导入采用批量事务（分批 commit）
   - 尽量避免长事务与大范围行锁

---

## 9. 稳定性与抗流量策略

1. **限流**
   - 搜索与对比接口按 IP / Token 限流
   - 推荐使用 Bucket4j（结合 Redis 计数）

2. **降级**
   - Redis 不可用时回落到 DB（并降低查询上限）
   - 搜索服务异常时返回空候选或本地兜底热门列表

3. **超时与重试**
   - DB/Redis 调用设置超时
   - 重试次数严格限制，避免雪崩放大

4. **错误隔离**
   - 导入错误不影响在线查询
   - 失败数据记录到 `import_task_error`

## 10. 可观测性与运维

### 10.1 指标（Metrics）
- API QPS、P95/P99 延迟
- 缓存命中率、回源次数
- 导入成功率、失败率、平均耗时
- 锁等待时间与获取失败次数

### 10.2 日志（Logging）
- 全链路 `traceId`
- 结构化 JSON 日志
- 导入任务按 `taskId` 关联日志

### 10.3 监控面板（Grafana）
- 接口性能总览
- 导入任务看板
- Redis/DB 健康状态

## 12. 开发排期（2~4 周）

### 第 1 周：项目骨架与基础查询
- 建立模块化单体结构（controller/service/repository）
- 完成 `player` 与 `search` 基础接口
- 建库建表与核心索引

### 第 2 周：导入与并发控制
- 完成 CSV/Excel 导入流程
- 接入导入任务表与错误明细表
- 加入分布式锁与幂等控制

### 第 3 周：对比与预计算
- 完成 `compare` 接口
- 增加每日预计算任务与结果表
- Redis 缓存 + 热点预热

### 第 4 周（可选强化）
- 接入 Prometheus/Grafana
- 压测与慢查询优化
- 补齐测试与面试文档

---

## 13. 亮点（单机版本）

> “将系统设计为模块化单体，在单机环境下通过预计算结果表、Redis 热点缓存、分布式锁与幂等机制，实现了低延迟查询和稳定导入；同时接入可观测体系，能够定位性能瓶颈并支撑中等规模并发访问。”

---

## 14. 后续演进路径（从单机到分布式）

1. `search` 模块独立为搜索服务（接入 ES）
2. `stats` 模块迁移到分析引擎（ClickHouse）
3. 导入链路从同步任务升级为消息队列（Kafka）
4. 单机数据库升级为主从与读写分离

> 演进原则：先保证当前可交付，再按瓶颈逐步拆分，不提前过度设计。

---

## 15. 结论


**低延迟、并发可控、稳定性、可观测、可演进**。  
对于当前项目阶段，这是比“直接上微服务”更务实、风险更低、成功率更高的路线。
