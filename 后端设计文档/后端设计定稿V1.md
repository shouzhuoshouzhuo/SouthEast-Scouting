# 后端设计定稿 V1（单机落地版）

> 定稿时间：2026-03-27  
> 适用阶段：MVP（2~4 周）  
> 部署形态：单机 Docker Compose

---

## 1. 定稿结论（本次确认）

- 架构形态：模块化单体（不做微服务拆分）
- 技术选型：**PostgreSQL 16 + Redis 7 + Spring Boot（Java 17）**
- 部署方式：**Docker Compose 单机部署**
- MVP 功能范围：
  - 球员搜索（姓名模糊 + 联赛/球队/位置筛选）
  - 球员详情
  - 1~3 人对比
  - 后台数据导入（CSV/Excel）
- 指标策略：**指标先不定稿**，后续根据 CSV 字段通过私有公式计算生成

---

## 2. 目标与范围冻结

### 2.1 业务目标

## 3. 指标设计说明（延后定稿）

### 3.1 原则

- V1 仅提供指标结果输出能力，不提前固定完整指标字典。
- 指标由“CSV 原始字段 + 私有公式”计算得到。
- 公式策略后续单独评审并版本化管理。

### 3.2 实施口径

- 首版导入阶段先落地原始数据与必要标准化字段。
- 预留指标快照表用于存储计算结果。
- 指标计算任务支持重跑（按赛季/联赛/批次）。

### 3.3 后续新增文档（建议）

- `指标公式设计V1.md`：定义公式、输入字段、边界条件、版本号。
- `指标校验样例集.md`：提供样例输入和期望输出用于回归测试。

## 4. 模块与数据设计（V1）

### 4.1 模块边界

- `player`：球员主档与详情聚合
- `search`：模糊搜索与筛选查询
- `compare`：1~3 人对比结果聚合
- `importing`：CSV/Excel 导入、校验、入库、任务追踪
- `jobs`：预计算与缓存预热任务

### 4.2 数据表（核心）

- `player`：球员基础信息
- `player_season_stat`：赛季原始统计
- `player_metric_snapshot`：指标计算结果快照（字段后续按公式定稿）
- `import_task`：导入任务状态
- `import_task_error`：导入错误明细

### 4.3 索引与性能要点

- `player.name` 使用 `pg_trgm` 支持模糊搜索
- `player(league, team, position)` 组合索引
- `player_season_stat(player_id, season)` 索引
- `player_metric_snapshot(player_id, season)` 唯一或高选择性索引

## 5. 缓存、并发与稳定性策略

### 5.1 Redis 使用场景

- 搜索热点结果缓存
- 对比结果缓存
- 导入任务互斥锁（`lock:import`）
- 限流计数（搜索/对比）

### 5.2 并发控制

- 同一时刻仅允许一个导入任务执行
- 文件哈希（SHA-256）做幂等校验，重复文件拒绝或标记已处理

### 5.3 降级策略

- Redis 不可用时查询回落 PostgreSQL，且降低分页上限
- 导入失败不影响在线查询功能

## 7. 环境与部署（确认版）

### 7.1 组件版本

- JDK 17
- Spring Boot（当前项目主版本）
- PostgreSQL 16
- Redis 7

## 9. 风险与待确认项

### 9.1 已知风险

- CSV 字段标准不统一会影响导入稳定性
- 指标公式未冻结前，对比口径可能变化
- 多赛季数据量增长可能影响查询性能

### 9.2 待确认事项

1. 指标公式的配置方式（代码内/配置文件/数据库）
2. 指标重算触发机制（手动/定时/导入后自动）
3. 首批必须支持的 CSV 模板字段清单

---

## 10. 结论

V1 方案已经冻结为“**单机 Docker Compose + PostgreSQL 16 + Redis 7**”路线，先确保搜索、详情、对比、导入四条核心链路闭环。指标体系作为下一阶段重点，通过“CSV 字段 + 私有公式”独立设计并版本化落地。
